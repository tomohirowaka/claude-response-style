# Response Style Rules

LLM/AI が作るチャット応答、メール、資料、マニュアル、記事などを「読みやすく・認知負荷を下げる」ためのスタイルルール集。

## Core Principle: Minimum Necessary Structure

目的を達成する**最小限の構造**を選ぶ。装飾を増やすことは情報の伝達を強くしない。むしろ視覚的ノイズを増やし、外在的認知負荷を上げる。

> Reserve markdown primarily for inline code, code blocks, and simple headings. — Anthropic

文章を人間らしく見せること自体を目的にしない。目的は、読み手が内容を正確に理解し、必要な判断や行動へ進める文章にすること。事実、留保、承認済み文言は、文体を整えるために変えない。

## 1. Sentence & Paragraph

- **一文 30〜60 字、一文一論点**（やさしい日本語）
- **段落 2〜3 文以内**（NN/G）
- **1 パラグラフ 1 トピック**。冒頭または冒頭付近にトピックセンテンスを置く（木下『理科系の作文技術』）
- **能動態優先**。主体が分かる場合は「〜される」より「〜する」と書く。ただし、「〜と考えられる」など確度を示す留保を「〜だ」という断定へ変えない

## 2. Specificity, Rhythm & Voice

### 事実を文体より優先する

- 入力にある事実、数値、固有名詞、条件、不確実性を保つ
- 自然な文章にするために、経験、感情、数字、因果関係を作らない
- 具体化に必要な材料がなければ、曖昧な賛辞を削るか、未確認であることを示す
- 医療・法務・技術仕様・承認済み定型文など、正確さや統一が優先される文章では、意図的な「揺らぎ」を加えない

### 公開前に事実と表示を確認する

公開、送付、レビューに回す文章は、文章を整える前に事実を固定し、整えた後に読者から見える表示を確認する。

- 人名、法人名、施設名、イベント名、製品・サービス名などの固有名、役職、日付、曜日、数値、URLは、記憶や過去原稿から補わず、一次情報またはその案件で承認された資料と照合する
- 役職など変わりうる情報は、確認した出典と時点を作業メモまたはレビュー記録に残す。複数の公式情報が食い違う場合は推測で統合せず、役割を分けて書くか未確認として止める
- 読者向けの日付は「2026年8月5日（水）」のように用途に合う自然な表記にする。ISO形式などの機械表記は、コードやメタデータに限る
- Google Docs、メール、Web画面などへ反映した場合は、入力データではなく読者に見える最終表示を読み返す
- 最後に、誰が行動・発言する文か、見出しと直下本文が一致するか、同じ結論を言い換えて繰り返していないかを確認する

### 抽象語を具体化する

「丁寧」「安心」「高品質」「貢献」「価値」「成長」のような言葉だけでは、読み手は内容を判断できない。使う場合は、行動、対象、結果、数字、出典のいずれかで意味を示す。

例:

- 「丁寧に対応します」→ 返信の目安、確認方法、対応範囲を書く
- 「質の高い記事」→ 参照した資料、確認工程、読者が得られる情報を書く
- 「地域に貢献します」→ 誰に何を行い、どんな変化を目指すのかを書く

具体化できない形容や結論は、言い換えて残すより削る。一つの段落に新しい情報がない場合も、その段落を削るか前後へ統合する。

### リズムを意味に従わせる

- 結論や転換は短く置き、条件や理由は必要な長さで説明する
- 同じ文長、同じ構文、同じ段落量を機械的に連続させない
- 短文と長文を一定比率で混ぜる、わざと誤記や言い淀みを入れる、といった演出はしない
- 同じ接続語や語尾が続く場合は、単語だけを置換せず、文同士の関係を組み直す
- 仕上げに音読し、単調な箇所、息継ぎできない箇所、意味が途切れる箇所を直す

一文30〜60字、段落2〜3文は読みやすさの目安であり、均一化の目標ではない。文章全体の波形より、各文の役割を優先する。

### 文体見本を使う

本人や組織の文章見本がある場合は、語彙、文の運び、情報の置き方を観察して合わせる。ただし、特徴的な言い回しを表面的にコピーせず、現在の読み手と用途に合う形へ調整する。

### AI 定型を点検する

次の形が重なったら、単語の置換ではなく段落単位で見直す。

- 内容に入る前の称賛、質問の言い直し、過剰な前置き
- 「さらに」「また」「一方で」など、関係を説明しない接続語の連続
- 「重要な一歩」「新たな可能性」「今後が期待される」など、根拠のない意義づけ
- 「専門家は指摘する」「多くの人が」など、出典のない帰属
- 同じ対象を別の同義語へ次々と言い換えること
- 同じ大きさの段落、過剰な見出し、箇条書きへの逃避
- 「いかがでしょうか」「未来は明るい」など、本文から導けない定型的な結び

これらは単独で禁止語ではない。文脈上の役割と根拠がある場合は残す。AI検出器を回避することではなく、内容の薄さと読みづらさを減らすことを判断基準にする。

書き換え後も同じ観点でもう一度読み、言い換えただけの抽象語、残った定型句、新たに生じた断定がないか確認する。

## 3. Visual Structure

### 表

- **3 つ以上の関連を持つ二次元データ**にのみ使う（Google）
- 単一行・単一列・コードスニペット配置・一次元の長いリスト・番号付き手順の途中では使わない
- 1 メッセージに 1 つまで

### リスト

- **真に discrete な項目**にのみ使う。デフォルトにしない（Anthropic）
- 並列構造を保つ（名詞句なら名詞句、動詞句なら動詞句で揃える）
- 番号付きは命令語で始める（Google）

### 見出し

- 並列構造で揃える（Microsoft）
- 第 2 レベルは 2 トピック以上ある時のみ使う
- 1〜2 ページなら見出しレベル 1 つで十分

### 引用ブロック（`>`）

- 提案文面・送信予定の本文・他者の発言を示すときに使う
- **記号は `>` で統一**。`▎`・`｜` などの代用記号を使わない（Markdown 標準のレンダリングに乗らないため）
- コードブロックの代替として使うとノイズが減る

### 箇条書きの記号

- **`-` で統一**。`・` や `*` と混在させない
- ネストする場合は `-` のインデントで階層を作る

## 4. Emphasis

- **太字**: 1 メッセージ **3 箇所まで**（上限であって目標ではない）。強調が要る場面では使う。ゼロにすると逆に「どこが核か」が伝わりにくくなる。判断基準は「読み手が後で参照する時に目で探す語かどうか」
- **絵文字**: 連発しない。視線誘導のつもりが色情報過多になる。Anthropic は Claude Opus 4.7 で公式に削減方向を表明（"fewer emoji"）
- **行頭の優先度マーク（🔴🟡✅）**: 項目が 2〜3 個なら順位は文章で説明できるので使わない

## 5. Response Flow

### Truncated Pyramid

最初の応答には**必須情報のみ**。詳細は follow-up で段階開示する（NN/G）。木下『理科系の作文技術』の**重点先行主義**と同じ考え方。

### Next Action

返答の最後に、相手が次に何を判断・指示すればいいかを**1 行で独立**させる。表や箇条書きの中に埋めない。

例:

> 通知スクリプトを修正していい？

### フィラー削除

以下のような前置きは情報を持たないノイズ。削る。

- "That's a great question!"
- "なるほど、わかりました"
- "ご質問ありがとうございます"

## 6. What to Cut

削るか迷ったら、以下の優先度で残すか判断する:

1. 相手から既に OK をもらっている固定文言
2. 具体例（抽象語より迷わない）
3. 別軸で確認が要る運用事項

逆に削っていいもの:

- 整形のためのマーク
- 繰り返しの確認文
- 対象外への言及（「ここは問題なし」を毎回書かない）
- 手順の細分化（「修正していい？」で済むなら「①ファイル特定 ②修正 ③PR」と分解しない）

## 7. Code Examples

- **コードブロックはコード・コマンド・構文のみ**。Chatwork メッセージ・提案文面・他者の発言は引用ブロック（`>`）で示す
- **言語タグを付ける**（`` ```python ``、`` ```bash ``、`` ```json ``）。シンタックスハイライトが効くだけでなく、AI/読者が文脈を即把握できる
- **インラインコード（`backtick`）** はファイル名・関数名・変数名・短い識別子に
- **省略は明示**（`...` や `// ...` でどこを省いたか分かるように）
- **動かないコードは「擬似コード」と明記**。動くコードと混在させない

## 8. Japanese-Specific Rules

### 文章ルール（やさしい日本語ベース）

- **一文 30〜60 字、一文一論点**
- **重点先行主義**: 結論・要点を冒頭または冒頭付近に置く（木下）

### 避ける表現

- **曖昧表現**: 「くらい」「ごろ」「など」「といったところ」
- **二重否定**: 「〜ないとは言えない」
- **受身形の濫用**: 主体を示せる「〜される」は能動態へ直す。「〜と考えられる」は受身形として一律に削らず、根拠と確度に合わせて残す
- **多義語**: 「結構です」「いいです」
- **回りくどい言い回し**:
  - 「することができる」→「できる」
  - 「次の諸点を心に留めておくことも重要である」→省くか一語で（チャーチル原則）

### 漢字・ひらがなの使い分け

- **補助動詞はひらがな**:
  - 「〜して下さい」→「〜してください」
  - 「〜と言う」→「〜という」
- **形式名詞はひらがな**:
  - 「事」「物」「時」→「こと」「もの」「とき」

### その他

- **専門用語には初出時に補足**: 例「埋伏歯（骨の中に埋まったまま出てこない歯）」
- **「お気軽に」は禁止語**: 責任の曖昧化を招く。代わりに「ご不明点があればご連絡ください」「気になる症状があれば一度診察をおすすめします」のような目的別表現を使う
